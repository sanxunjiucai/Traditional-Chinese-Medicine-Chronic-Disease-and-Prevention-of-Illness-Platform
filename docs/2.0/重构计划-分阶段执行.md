# 中医慢病与治未病平台 2.0 重构计划

## 重构原则
1. **渐进式**：不推翻现有系统，逐步增强
2. **最小化**：每个阶段只做必要的核心功能
3. **可验证**：每个阶段都有明确的验收标准
4. **向后兼容**：保留现有 API，新增 v2 API

---

## 阶段 3：核心 API 工具（3-4天）

### 3.1 必须新增的 API（按业务流程）

#### 诊中流程 API
```
GET  /tools/v2/context/current              # 获取当前上下文
GET  /tools/v2/patients/{id}/summary        # 获取临床摘要
POST /tools/v2/patients/{id}/supplement     # 补充采集
POST /tools/v2/economic-assessment          # 经济评估
```

#### 判断与方案 API
```
POST /tools/v2/assessment/tcm               # 生成体质判断
POST /tools/v2/assessment/tcm/confirm       # 确认体质判断
POST /tools/v2/assessment/risk              # 生成风险判断
POST /tools/v2/assessment/risk/confirm      # 确认风险判断
POST /tools/v2/plans                        # 生成方案草稿
GET  /tools/v2/plans/{id}/preview           # 预览方案
POST /tools/v2/plans/{id}/confirm           # 确认方案
POST /tools/v2/plans/{id}/distribute        # 分发方案
```

#### 随访与召回 API
```
POST /tools/v2/followup/create-from-plan    # 从方案创建随访
POST /tools/v2/recall/evaluate              # 评估召回
POST /tools/v2/recall/{id}/accept           # 接受召回
```

### 3.2 验收标准
- [ ] 所有 API 有 OpenAPI 文档
- [ ] 请求/响应 schema 完整
- [ ] 错误码统一
- [ ] 幂等性保证

## 风险与应对

### 风险 1：HIS 集成复杂度未知
**应对**：先做独立插件，后期再对接 HIS

### 风险 2：AI 编排稳定性
**应对**：先用规则引擎，逐步引入 AI

### 风险 3：三端分发失败处理
**应对**：实现重试机制和部分成功处理
